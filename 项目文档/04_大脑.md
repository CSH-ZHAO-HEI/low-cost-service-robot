# 大脑（云端 LLM/VLM 规划与自验证层）

> 逐一核对自 `big_brain/`：`config.py`、`big_brain.py`、`model/llm.py`、`model/rag.py`、
> `action/robot_api.py`、`prompt/task_prompt.py`、`ros_bridge.py`、`vlm_image_writer.py`。

大脑 = 纯 Python（非 ROS 包），通过 `ros_bridge` 调用小脑。核心流程：
**RAG 检索 → Planner(DeepSeek) 生成 Python 脚本 → exec → 每个原语后 JudgeLLM（规则+VLM）判定 → 失败则 replan**。

## 2. 主流程（`big_brain.py::run_once`）

1. `rag_manager.retrieve(instruction)` 检索最相似历史成功任务。
2. 组装 `final_prompt = BASE_PROMPT + RAG 参考（带 "FOR INSPIRATION ONLY" 边界）+ 指令`。
3. `planner.generate_code()` → DeepSeek 生成 Python 脚本，`extract_code` 清洗。
4. **canonical helper 覆盖**：若指令命中 C1–C4 关键词，直接换成 `run_cX_*()`（helper 模式）；
   设环境变量 `BIGBRAIN_DISABLE_C_HELPER_CANONICAL=1` 关闭（**no-helper 实验用**）。
5. 执行前强制：`ros_bridge.restore_arm_clamp()`（手臂归位）+ `ros_bridge.clear_costmaps()`。
6. `exec(generated_code, exec_globals)`，注入 objects（L2 字典）+ 原语 + 查询工具。
7. 成功 → 写回 `memory/rag_history.json`（供后续 RAG）。

## 3. 底层行动原语（`action/robot_api.py`，757 行）

### 6 个基础原语（与 JudgeLLM action_id 一一对应）

| action_id | 原语 | 说明 |
|-----------|------|------|
| 1 | `move_to_xy(x, y)` | 导航到坐标（TEB） |
| 2 | `move_to_obj_by_offset(obj, dx, dy, extra_forward=0)` | 导航到物体 approach 点 + 偏移 |
| 3 | `pick_up_xy(x, y)` | 按坐标抓取 |
| 4 | `pick_up_obj(item, extra_forward=0)` | 按名称抓取（地面 z=0.025 / 桌面 z=0.4+ 切换姿态） |
| 5 | `put_down_xy(x, y, surface_z=0)` | 按坐标放置 |
| 6 | `put_down_obj_by_offset(target, dx, dy, extra_forward=0)` | 按物体偏移放置 |

每个原语内部调用 `judge_llm.judge(action_id=…, …)` 触发判定闭环。

### 任务级封装（helper，no-helper 实验时不暴露给模型）

`pick_up_from_coffeetable`、`pick_up_coke_from_balconytable`、`put_down_between_objs`、
`put_down_coke_on_nightstand`、`run_c1_coke_to_nightstand`、`run_c2_red_block_between_blue_yellow`、
`run_c3_all_ground_blocks_to_trash`、`run_c4_sofa_patrol_red_to_trash`。

### helper / no-helper 运行口径

| 口径 | 代码入口 | 作用 |
|------|----------|------|
| helper | `big_brain.py` 中的 canonical helper 覆盖 | 命中 C1-C4 关键词时，可直接调用稳定封装 |
| no-helper | `BIGBRAIN_DISABLE_C_HELPER_CANONICAL=1` | 关闭 canonical helper，要求模型从基础原语组合任务 |
| prompt no-helper | `experiments/C1-C4/run_e*_composite.py --no-helper-prompt` | 从 `BASE_PROMPT` 中移除 C1-C4 helper 示例和 import 提示 |

工程运行可使用 helper 提高稳定性；长程复合任务评估使用 no-helper 口径，重点观察模型是否能从 `move/pick/put` 等基础原语组合出循环、巡逻中断与恢复流程。

## 5. 三层自验证闭环（`model/llm.py::JudgeLLM`）

### 流程（`judge()`）

```
解析 action_id+params → 收集观测(robot_x/y, orientation, holding)
  → 规则层 _rule_judge（按 action_id 算几何误差/holding）
  → 若规则失败 或 action_id∈{3,4,5,6}（语义敏感）→ 调 VLM judge_VLM
  → result = rule_pass AND (不需VLM 或 vlm_pass)
  → pick/put 无可靠夹爪传感器 → 允许 VLM 覆盖规则初判（vlm_override）
  → 失败 → replan()
```

### 规则层判定（`_rule_judge`，阈值见 §1）

| action_id | 判据 |
|-----------|------|
| 1 move_to_xy | xy 误差 ≤ MOVE(0.25) |
| 2 move_to_obj | 到 approach 点误差 ≤ APPROACH(0.35) |
| 3/4 pick | holding=True **且** 离目标 ≤ CATCH(0.55) |
| 5/6 put | holding=False **且** 离目标 ≤ CATCH(0.55) |

### VLM 层（`judge_VLM`）

- 读两张图（`vlm_image_writer.py` 持续写）：
  - `image/image.jpg` ← **`/judge_camera/rgb/image_raw`**（车载侧拍）
  - `image/trash_inside.jpg` ← `/trash_inside/cam/image_raw`（桶内俯拍）
- system prompt = `prompt/vlm_prompt.py::VLM_SYSTEM_PROMPT`；只给 raw numerics（不给 failure_code 标签，避免 anchor）；
  put 动作明确告知「Currently Placing 哪个物件」防 false positive；temperature=0.1 输出稳定 JSON。

### 多口径判定

| 口径 | 来源 | 用途 |
|------|------|------|
| strict | 脚本最终 `final_result` / `e*_final_result` | 受 JudgeLLM 与 VLM 最终结论影响 |
| Gazebo | `gazebo/get_model_state`、物体 z 高度、桌面 bbox、两物体连线关系 | 作为仿真真值判定执行结果 |
| semantic | 附录统计脚本中的任务语义规则 | 避免把区域目标误收窄为单个固定坐标 |
| VLM | `judge_VLM` 输出 JSON | 视觉语义确认，可辅助抓取、放置和容器任务 |

G3/G4 和 C 组任务会同时记录这些口径。这样可以区分“机器人动作没有完成”和“视觉判定较严格”两类情况。

### 重规划层（`replan`）

- **Per-block 配额**：red 的 pick+move+put 共用一个 counter；用光 → 标记 `dead_blocks`，
  后续对该 block 的动作直接 PASS（让 plan 进到下一个 block，不卡死整任务）。
- 注入：机器人当前状态 + 工具清单 + 最近 4 条失败历史 → `ADJUST_PROMPT` → DeepSeek 生成恢复流程。
- `_add_extra_forward_for_close_retry`：VLM 说「move closer/grasp_missed」时，AST 自动给
  `pick_up_obj`/`put_down_obj_by_offset` 加 `extra_forward` 参数。
- `REPLAN_EXECUTE_CODE=True` → 执行恢复流程（内部又会触发 judge，靠 MAX_REPLAN_TIMES 防无限递归）。
- 全程写 `outputs/judge_logs/judge_events.jsonl`（实验证据）。

### 实验中的重规划次数

| 场景 | 代码参数 | 说明 |
|------|----------|------|
| 默认配置 | `MAX_REPLAN_TIMES=1` | `config.py` 中的缺省值 |
| E1 主实验 | `adjust_policy=2` | N/G/C runner 可在运行时覆盖 JudgeLLM 配额 |
| E2 消融 | `adjust_policy=0/1` | 对比不恢复与允许一次恢复 |
| E4 基线 | `adjust_policy=0` | 人工 API 序列，不调用 LLM/VLM 重规划 |

`adjust_policy` 是实验脚本层面的覆盖参数，不改变 `config.py` 的缺省值。调用次数越多，恢复机会越多，但也会增加云端调用、VLM 判定和总执行时间。

## 6. RAG（`model/rag.py`）

- `SentenceTransformer(all-mpnet-base-v2)` 对历史 command 批量编码。
- 新指令 → `util.cos_sim` 逐条比对 → 取最高分；≥0.5 则把该历史的 `command` + `task_queue` 作参考注入。
- 消融结果：C3（多物体循环）no-helper 下关 RAG 可执行率 0 → 开 RAG 约 78%。

## 7. 提示词（`prompt/task_prompt.py::BASE_PROMPT`）

四层结构：① 单位约定（全 m，50cm=0.5）② 接口声明（import 原语）③ 固定示例（坐标导航/按名抓取/桌面放置/两物间放置/方形巡逻/循环）④ 动态 RAG 片段。
其余 prompt：`object_prompt.py`（对象解析）、`position_prompt.py`、`vlm_prompt.py`、`adjust_prompt.py`（replan）。
