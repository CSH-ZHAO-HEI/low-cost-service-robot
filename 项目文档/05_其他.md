## 1. 抓取系统（arm_task_server + block_follower_plugin）

### arm_task_server.py — 机械臂任务服务

5 个 ROS Service（`std_srvs/Trigger`）：

| Service | handler | 作用 |
|---------|---------|------|
| `/arm/pick` | handle_pick | 抓取 `/arm_task/target_name` 指定物件 |
| `/arm/put` | handle_put | prepare_put + drop 合并 |
| `/arm/prepare_put` | handle_prepare_put | 抬臂到 ARM_PUT，follower 继续跟随（方块随臂转向） |
| `/arm/drop` | handle_drop | 停 follower → 开爪 → teleport 方块到目标面 |
| `/arm/home` | handle_home | 归位 ARM_CLAMP |

**姿态常量（实测值）**：

```python
ARM_CLAMP       = [0.0, -1.1, 0.66, 1.0, 0.0]   # 搬运/初始
PICK_JOINTS     = [0.0, -1.3, 0.66, 1.0, 0.0]   # 地面抓取深度
ARM_PUT         = [0.0, -0.5, 0.3,  0.3, 0.0]   # 放置（前上 45°）
ARM_PICK_HIGH   = [0.0, -0.5, 0.3,  0.3, 0.0]   # 桌面抓取（=ARM_PUT）
PICK_HIGH_LOWER = [0.0, -0.6, 0.5,  0.5, 0.0]   # 桌面下降到物件
GRIP_OPEN  = [ 0.45, -0.45,  0.45,  0.45,  0.45,  0.45]
GRIP_CLOSE = [-0.45,  0.45, -0.45, -0.45, -0.45, -0.45]
BLOCK_HALF=0.025  GRIP_Z_OFFSET=-0.03  GRIP_XY_OFFSET=0.0
```

**地面 / 桌面 pick 切换**：`handle_pick` 读 `/arm_task/source_z`，
≤0.15 → 地面（ARM_CLAMP→PICK_JOINTS）；>0.15 → 桌面（ARM_PICK_HIGH→PICK_HIGH_LOWER）。

**Z_HEIGHTS 备用桌面高度**（yaml 无 surface_z 时模糊匹配，无匹配回 0.025 落地）。

### block_follower_plugin.cpp — C++ 搬运插件

| 项 | 说明 |
|----|------|
| 类型 | Gazebo World Plugin，`ConnectWorldUpdateBegin`（物理步频，~1000Hz） |
| `/block_follower/attach` | `std_msgs/String` `"robot::link,block,x,y,z"` 开始跟随 |
| `/block_follower/detach` | 任意字串，停止跟随 |
| 原理 | attach 时算方块相对 link5 本地坐标系的 offset；OnUpdate 内 `SetWorldPose` 让方块零延迟跟随手臂 |
| detach 后 | 持续 `SetEnabled` 唤醒，对抗 ODE auto-disable |

> 这是 pick 流程使用的搬运机制（绕过 ROS round-trip 延迟）。URDF 另声明的
> `gazebo_grasp_fix` 插件（见《03_车辆本体》§4）并存，但搬运由本插件负责。

### 夹爪中心计算

`center = (link7 + link9)/2 + GRIP_Z_OFFSET(-0.03)`，来源 `/link_states` 缓存。

## 2. 场景抽取（get_scene.py → L2 语义-几何资料库）

```bash
python3 scripts/get_scene.py               # 输出 gazebo_scene.yaml
python3 scripts/get_scene.py --deploy      # 同时写 ~/.ros/semantic_map.yaml（大脑用）
python3 scripts/get_scene.py --approach 0.4  # 指定接近距离（默认 0.35m）
```

> 输出路径硬编码：`OUTPUT_PATH=<PROJECT_ROOT>/gazebo_scene.yaml`、`DEPLOY_PATH=~/.ros/semantic_map.yaml`。

- `get_model_bbox`：trimesh 读 SDF collision mesh → `surface_z = bbox_max_z + collision_pose_z + link_pose_z`，再加模型世界 z。
- `compute_approach` / `compute_all_approaches`：按 yaw 算 oriented bbox 边界，approach 点落在 bbox 外侧；4 个世界方向（E/W/N/S）择可达且近者。
- `compute_place_point`：放置点落在桌面/桶口内侧（区别于 bbox 外的导航 approach 点）。
- 每条记录字段：name、坐标、颜色、尺寸、approach_x/y/yaw、place_x/y、surface_z。

## 5. 实验（`experiments/`，对应实验评估 E1–E4）

三组任务 × 四组实验：

| 组 | 任务 | 关键脚本 | 结果 |
|----|------|---------|------|
| **N1-N4** 导航 | nav_capability_test.py | run_flash_e1e2e3_nav.py、run_pro_e3_nav.py、run_e4_nav.py | outputs/E1~E3_flash.csv、E4.csv |
| **G1-G4** 抓取 | manip_capability_test.py、composite_capability_test.py | run_e1/e3/e4_manip.py、test_e2_vlm_manip.py | outputs/G-E1~E4.csv |
| **C3-C4** 复合主评估 | composite_capability_test.py | run_e1/e2/e3/e4_composite.py、run_c3_red_only.py | outputs/C-E1~E4.csv（含 no-helper 版） |

**评估口径**：E1 成功率基准 / E2 闭环介入纠偏 / E3 静态规划评估 / E4 人工 API 基线与开销。
**消融**：RAG on/off、no-helper（从零规划）vs helper（兜底）。no-helper 由
`BIGBRAIN_DISABLE_C_HELPER_CANONICAL=1` 关闭 canonical helper 触发（见《04_大脑》§2）。

`experiments/C1-C4` 中保留 C1、C2 runner 和变体脚本，便于验证桌面抓取、可乐罐搬运和两物体之间放置等局部能力；主复合任务不重复统计 C1/C2，因为这些能力已经由 G3、G4 覆盖。C 组主评估聚焦 C3 多物体循环和 C4 巡逻中断恢复。

> 核心实验结论：C3 多物体复合任务 no-helper 下，RAG 关闭可执行率 0 → 开启约 78%；
> 复合任务首次执行多失败，但多数可由「规则+VLM+重规划」三层闭环恢复。
